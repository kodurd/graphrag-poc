Q1: What is being changed about consumer group rebalance behavior?  | route: mixed
A: На основе предоставленного контекста можно выделить следующие изменения в поведении ребалансировки потребительских групп:

1. **Логирование рекомендации по переходу на новый протокол ребалансировки**  
   Для классического потребителя (Classic consumer) добавлено логирование, рекомендующее использовать новый протокол ребалансировки, представленный в KIP-848 (доступен в Apache Kafka 4.0). Это первая задача KIP-1274, а дальнейшие изменения (смена протокола по умолчанию, устаревание и удаление классического протокола) запланированы на AK 5.0 и 6.0 [источник: https://issues.apache.org/jira/browse/KAFKA-20282].

2. **Устранение двойной ребалансировки при перезапуске реплик (rack-aware)**  
   В классическом потребителе при перезапуске реплики (replica bounce) происходила двойная ребалансировка из-за временного изменения rack ID (с `rackX` на `null`, затем обратно). Предлагается избежать этого, определяя, что реплика временно недоступна (потеряла "host"), а не только rack. В новом протоколе потребителя такой проблемы нет [источник: https://issues.apache.org/jira/browse/KAFKA-20778].

3. **Проблема липкости (stickiness) при запуске для групп "Streams"**  
   Для групп, использующих протокол "Streams" (KIP-1071), липкий ассайнер (sticky assignor) не работает при запуске нового экземпляра/клиента. В классическом протоколе новый клиент передаёт информацию о найденных локальных директориях задач, что позволяет ассайнеру сохранять привязку задач к предыдущим клиентам. В "Streams" группах такая информация не кодируется, и новый член группы получает новый memberId, из-за чего предыдущее состояние теряется [источник: https://issues.apache.org/jira/browse/KAFKA-20719].

4. **Фоновые heartbeat для предотвращения выпадения из группы**  
   Предлагается отправлять heartbeats из фонового потока, чтобы потребитель не выпадал из группы из-за длительной обработки данных. Однако если ребалансировка начинается во время обработки, потребитель всё ещё может не успеть присоединиться к группе, особенно при модели пакетной записи в удалённую систему [источник: https://cwiki.apache.org/confluence/spaces/KAFKA/pages/63406974/KIP-62+Allow+consumer+to+send+heartbeats+from+a+background+thread].

5. **Общее описание ребалансировки**  
   Ребалансировка — это процесс, в котором экземпляры потребителей одной группы координируются для взаимного исключения владения разделами подписанных топиков. Координатор группы (брокер) управляет ребалансировкой при изменениях членства в группе или разделов, а затем сообщает конфигурацию владения разделами всем потребителям [источник: https://cwiki.apache.org/confluence/spaces/KAFKA/pages/38570548/Kafka+0.9+Consumer+Rewrite+Design].

**Вывод:** Основные изменения касаются логирования рекомендаций, устранения двойной ребалансировки, улучшения липкости для "Streams" групп и внедрения фоновых heartbeat. Детали реализации некоторых изменений (например, KIP-1274) будут уточнены в будущих версиях.

Q2: How is Kafka Connect handling exactly-once or duplicate records?  | route: mixed
A: На основе предоставленного контекста, Kafka Connect по умолчанию использует семантику **at‑least‑once** (как минимум один раз). Поддержка **exactly‑once** (ровно один раз) возможна только в ограниченном сценарии: для Kafka‑to‑Kafka в одном кластере, когда доступны транзакционные подтверждения разделов (share acknowledgments) согласно KIP‑1289 [источник: https://issues.apache.org/jira/browse/KAFKA-20367].

Дублирование записей может возникать, например, из‑за ошибок при восстановлении смещений после перезапуска задачи, что приводит к повторной обработке [источник: https://issues.apache.org/jira/browse/KAFKA-20666]. В контексте также упоминается, что Kafka предоставляет семантику "at least once", и дубликаты могут появляться при повторных попытках отправки после сетевых ошибок [источник: https://cwiki.apache.org/confluence/spaces/KAFKA/pages/38571308/Idempotent+Producer].

Дополнительных механизмов для обработки дубликатов (например, дедупликации на стороне коннектора) в предоставленном контексте не описано.
